# Prompt 模式：代码重构

## 使用场景
用户需要重构已有代码（改善结构但不改变外部行为）时使用此模式。

## 必须提供的上下文
1. 重构目标（解决什么问题）
2. 重构范围（哪些模块/类）
3. 约束条件（不能改变哪些行为）
4. 是否有测试覆盖

## Agent 工作流程

### Step 1: 现状分析
- 理解当前代码结构
- 识别存在的问题（命名、职责、耦合、复杂度）
- 评估改动范围和风险

### Step 2: 重构方案
- 列出具体的重构操作
- 说明每步操作的原因
- 确认不改变外部行为
- 提交用户确认

### Step 3: 分步实现
重构必须分步进行，每步保证可编译、可运行：
1. 提取/重命名/移动 → 编译验证
2. 调整依赖关系 → 编译验证
3. 简化逻辑 → 测试验证

### Step 4: 验证
- 现有测试全部通过
- 补充缺失的测试
- 接口行为不变

## 重构原则
- 每步只做一种类型的重构
- 重构与功能变更不混在一个 PR
- 先补测试再重构
- 保持向后兼容

## 常见重构类型及方法
| 问题 | 重构方法 |
|------|---------|
| 类过大 | 提取类 / 拆分职责 |
| 方法过长 | 提取方法 |
| 重复代码 | 提取公共方法/基类 |
| 过度耦合 | 引入接口/依赖注入 |
| 命名不清 | 重命名 |
| 层次混乱 | 调整分层 |

- `07-backend-architecture.mdc` — 分层架构
