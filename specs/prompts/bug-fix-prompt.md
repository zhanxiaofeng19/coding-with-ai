# Prompt 模式：Bug 修复

## 使用场景
用户报告 Bug 或需要修复已知问题时使用此模式。

## 必须提供的上下文
1. Bug 现象描述
2. 复现步骤（如有）
3. 影响范围
4. 是否线上问题
5. 相关日志/错误信息

## Agent 工作流程

### Step 1: 问题分析
- 理解 Bug 现象
- 定位可能的根因
- 确认影响范围
- 判断严重级别（P0~P3）

### Step 2: 根因定位
- 阅读相关代码
- 分析调用链路
- 检查异常处理逻辑
- 检查并发安全
- 检查边界条件

### Step 3: 方案设计
如果修复 > 20 行代码：
- 说明根因
- 描述修复方案
- 分析修复对其他逻辑的影响
- 提交用户确认

如果修复 ≤ 20 行代码：
- 直接说明根因和修复方案
- 可合并设计与实现

### Step 4: 修复实现
- 修复问题代码
- 补充防御性代码
- 补充相关日志

### Step 5: 自检
- 按 `04-self-review.mdc` 检查修复代码
- 确认修复不引入新问题

### Step 6: 测试
- 补充覆盖该 Bug 的单元测试
- 验证修复有效

### Step 7: 输出修复报告
- 根因分析
- 修复方案
- 影响范围
- 预防措施

## 线上 Bug 特殊流程
如果是线上紧急 Bug：
1. 优先止血（降级/回滚/配置变更）
2. 再定位根因
3. 修复 + 测试
4. 灰度发布
5. 输出复盘报告

## 必须遵循的规范
- `04-self-review.mdc` — 自检
- `05-git-workflow.mdc` — fix 分支规范
