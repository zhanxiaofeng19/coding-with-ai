# 工程原则

## 核心信条

### 1. 简单优于复杂
- 用最简单的方案解决问题
- 不为"将来可能需要"提前设计
- 每引入一层抽象必须有明确理由

### 2. 可维护性优于性能
- 性能优化只在有数据支撑时进行
- 代码可读性优先
- 事故压力下任何人都能读懂代码

### 3. 生产思维优先
- 默认假设依赖会失败
- 默认假设流量会突增
- 默认假设数据会不一致
- 每段代码都要回答"出问题了怎么办"

### 4. 最小改动原则
- 只改需求要求的部分
- 不顺手重构不相关代码
- 接口只增不改
- 保持向后兼容

### 5. 可观测性是必需品
- 不可观测的系统等于不可运维
- 日志、指标、链路追踪三位一体
- 关键操作必须留痕

## 设计决策框架

当面临技术选型时，按以下优先级决策：

1. 正确性 — 结果必须正确
2. 可用性 — 服务必须可用
3. 可维护性 — 代码必须可维护
4. 性能 — 满足 SLO 即可
5. 扩展性 — 按当前需求设计，预留扩展点
